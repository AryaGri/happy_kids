## 2.1 Анализ технического задания и выбор средств разработки

Техническое задание на разработку программной системы предполагает создание веб-приложения и десктопного клиента для оценки когнитивных и эмоциональных процессов у детей дошкольного и младшего школьного возраста с использованием игровых методик и методов нечёткой логики. Ниже приведён обоснованный выбор технологического стека.

**Ядро серверной части.** В качестве веб-фреймворка выбран **Django** (версия 4.2 и выше). Причина выбора: встроенная ORM для работы с базой данных, система миграций, механизм сессий и аутентификации, развитая экосистема расширений и соответствие требованиям масштабируемости. Django обеспечивает строгую структуру приложения (MVC/MVT), что облегчает сопровождение и развитие системы.

**База данных.** В режиме разработки используется **SQLite** — встроенная СУБД, не требующая отдельной установки; для production предусмотрена **PostgreSQL** через расширение dj-database-url. Выбор обусловлен надёжностью, поддержкой транзакций и JSON-полей, необходимыми для хранения траекторий поведения и нечётких показателей.

**Математические расчёты.** Библиотека **NumPy** применяется для вычисления средних, стандартных отклонений, коэффициентов корреляции и трапециевидных функций принадлежности. NumPy обеспечивает эффективную работу с массивами, что критично при обработке серий времени реакции и эмоциональных метрик.

**Обработка изображений.** **Pillow** используется для сохранения рисунков ребёнка (игра «Раскраска»), генерации превью и работы с медиафайлами. Выбор Pillow обусловлен широкой поддержкой форматов и интеграцией с Django (ImageField).

**Генерация PDF.** Библиотека **ReportLab** применяется для формирования PDF-отчётов с назначениями врача для родителей. ReportLab позволяет создавать структурированные документы с поддержкой кириллицы и произвольной версткой.

**Развёртывание.** Для production-режима используются **Gunicorn** (WSGI-сервер) и **WhiteNoise** (раздача статических файлов без отдельного веб-сервера). Данная связка обеспечивает высокую производительность и простоту деплоя на облачных платформах (Render.com).

**Десктопный клиент.** Приложение реализовано на **CustomTkinter** и **PyWebView**. CustomTkinter предоставляет современный графический интерфейс для экрана настройки (адрес сервера) и быстрого запуска. PyWebView отображает веб-интерфейс системы в нативном окне, что позволяет использовать единый код фронтенда (HTML/CSS/JS) для веб- и десктопной версий без дублирования логики.

**HTTP-клиент.** Библиотека **Requests** используется в десктопном клиенте для проверки доступности сервера и в API-взаимодействии. Выбор обусловлен простотой использования и надёжностью при работе с REST API.

**Визуализация.** На стороне веб-интерфейса применяются **Chart.js** (графики эмоций, радарные диаграммы), **Bootstrap 5** (адаптивная разметка) и **Font Awesome** (иконки). На стороне десктопного клиента — **Matplotlib** для построения графиков при необходимости.

## 2.3 Разработка структуры программной системы

Структура системы построена по модульному принципу с чётким разделением ответственности.

**1. Веб-сервер (Django).** Приложение `accounts` содержит модели данных, представления (views), формы, модуль нечёткой логики и диагностической панели. Маршрутизация настроена в `accounts/urls.py` и подключается через `my_project/urls.py`. Конфигурация проекта вынесена в `my_project/settings.py`.

**2. Модуль нечёткой логики.** Отдельный модуль `accounts/fuzzy_logic.py` реализует классы FuzzySet, FuzzyVariable и FuzzyAnalyzer, обеспечивающие фаззификацию, анализ и формирование рекомендаций. Модуль `accounts/diagnostic_panel.py` дополняет визуализацию и интерпретацию диагностических параметров.

**3. REST API.** Подмодуль `accounts/api_views.py` предоставляет эндпоинты для десктопного клиента: аутентификация по токену, получение списка пациентов, детальной информации о пациенте, создание назначений. API использует заголовок `X-Auth-Token` для авторизации врачей.

**4. Десктопный клиент.** Каталог `desktop_client/` содержит:
- `app.py` — единая точка входа: экран настроек (CustomTkinter), проверка соединения, запуск PyWebView с веб-интерфейсом;
- `main.py` — обёртка для запуска через bat-файл;
- клиенты по ролям (`admin_client.py`, `doctor_client.py`, `parent_client.py`, `child_client.py`) — дополнительные утилиты при необходимости.

**5. Игровой модуль.** Каждая игра реализована отдельным представлением (view) и шаблоном. Результаты сохраняются в `GameResult` и `GameSession`. Передаваемые метрики: эмоции (joy, sorrow, love, anger, boredom, happiness), время реакции (reaction_times), количество ошибок (mistakes), performance_metrics (специфичные для типа игры).

**6. Деплой.** Конфигурация для Render.com задаётся в `render.yaml`. Сборка десктопного приложения выполняется через `build_desktop.bat` (PyInstaller, platform.spec).

## 2.4 Проектирование базы данных

Схема базы данных построена на моделях Django ORM. Ниже приведено описание основных таблиц и их назначение.

### 2.4.1 Основные сущности

**CUsers** (Пользователи). Центральная таблица пользователей с полями: username, name, date_of_b, role (admin/doctor/parent/child), password (хешированный), is_auth, created_at. Связи: ManyToMany children (родитель–дети), ManyToMany patients (врач–пациенты). Дополнительно: connection_code, code_expires — код для подключения ребёнка к родителю/врачу.

**DoctorLicense** (Лицензии врачей). Связь OneToOne с CUsers. Поля: license_number, license_file, license_scan, is_verified, verified_by (FK на администратора), verified_date, rejection_reason.

**GameSession** (Игровые сессии). Привязка к пользователю (user), тип игры (game_type), start_time, end_time, completed. Поле behavior_trajectory (JSONField) хранит историю действий для анализа поведенческих траекторий.

**GameResult** (Результаты игр). Основная таблица для хранения данных игр. Связи: user, session. Поля:
- эмоции: joy, sorrow, love, anger, boredom, happiness;
- время реакции: reaction_time, reaction_times (JSON);
- ошибки: mistakes, mistake_types (JSON);
- подсказки: hints_used, hint_timing (JSON);
- strategy_type, accuracy, performance_metrics (JSON), fuzzy_analysis (JSON);
- данные игр: drawing_data, final_image (Painting), dialog_answers (Dialog), choices (Choice).

**Prescription** (Назначения). Связи: child, doctor. Поля: text, prescription_type (medication/therapy/exercise/recommendation), medication_name, dosage, duration, date_created, is_active.

### 2.4.2 Модели нечёткой логики

**FuzzyLinguisticVariable.** Лингвистические переменные с полями name, description, terms (JSON — словарь термов и параметров функций принадлежности), purpose.

**FuzzyMembershipFunction.** Функции принадлежности для классов методов (traditional_test, projective, observation, digital_game, hybrid). Связь с FuzzyLinguisticVariable, membership_values (JSON).

**DiagnosticProfile** (Диагностический профиль). Связь с ребёнком (child). Поля: diagnostic_depth, motivational_potential, objectivity, ecological_validity, dynamic_assessment (все JSON с принадлежностями к термам), cognitive_style (systematic/impulsive/adaptive/chaotic), emotional_profile (JSON), recommendations, detected_diagnoses (JSON — коды), based_on_sessions (M2M с GameSession).

**DiagnosticDiagnosis** (Диагностические диагнозы). Справочник с полями: name, code, fuzzy_conditions (JSON — условия по нечётким показателям), default_recommendations, default_prescription_type, default_prescription_text, priority.

**FuzzyInferenceRule** (Правила нечёткого вывода). Поля: name, condition (JSON), conclusion, recommendation, confidence.

**BehaviorPattern** (Поведенческие паттерны). Поля: name, pattern_type (strategy/attention/emotion/social), description, fuzzy_sets (JSON), relevant_games (JSON).

### 2.4.3 Дополнительные сущности

**Subscription** (Подписки). Связь OneToOne с пользователем. Типы подписок: parent_individual, parent_family, psychologist_individual, clinic_small/medium/large, kindergarten. Параметры: max_children, max_doctors, start_date, end_date, is_active, price.

**ApiToken** (API-токены). Связь с пользователем, поле token. Используется для авторизации десктопного клиента и мобильных приложений.

## 2.5 Разработка алгоритмов оценки с использованием нечёткой логики

### 2.5.1 Фаззификация входных игровых данных

Фаззификация представляет собой преобразование числовых показателей игр в степени принадлежности к лингвистическим термам. Реализация выполнена в классе `FuzzySet` с трапециевидными функциями принадлежности μ(x) для параметров [a, b, c, d].

**Входные переменные и их термы:**

1. **Диагностическая глубина** (диапазон 0–1):
   - низкая: [0, 0, 0.3, 0.4];
   - средняя: [0.3, 0.5, 0.7, 0.8];
   - высокая: [0.7, 0.8, 1, 1].

2. **Мотивационный потенциал** (0–1):
   - низкий: [0, 0, 0.2, 0.3];
   - умеренный: [0.2, 0.4, 0.6, 0.7];
   - высокий: [0.6, 0.8, 1, 1].

3. **Объективность и стандартизация** (0–1):
   - низкая, средняя, высокая (аналогичная структура термов).

4. **Экологическая валидность** (0–1):
   - низкая, средняя, высокая.

5. **Потенциал для динамической оценки** (0–1):
   - ограниченный, умеренный, широкий.

6. **Уровень импульсивности** (время реакции в мс):
   - низкий: [0, 0, 300, 400];
   - средний: [300, 400, 600, 700];
   - высокий: [600, 800, 2000, 2000].

7. **Познавательная активность** (частота подсказок, шкала 0–10):
   - исследующая, направленная, зависимая.

Интегральные оценки для лингвистических переменных A–E вычисляются на основе:
- полноты данных по каждому типу игры (`_game_data_score`);
- разнообразия типов игр (diversity);
- количества сессий (sessions_score);
- доли объективных игр (Memory, Puzzle, Sequence, EmotionFace и др.);
- доли субъективных игр (Painting, Dialog, Choice);
- показателей прогресса (levels_completed, level_reached, completed).

Пример метода фаззификации для времени реакции:

```python
def analyze_reaction_times(self, reaction_times: List[float]) -> Dict[str, float]:
    avg_time = np.mean(reaction_times)
    cv = np.std(reaction_times) / avg_time if avg_time > 0 else 0
    impulsivity = self.variables['impulsivity'].fuzzify(avg_time)
    if cv > 0.3:  # высокая вариабельность
        impulsivity['высокий'] = min(1, impulsivity['высокий'] + 0.2)
    return impulsivity
```

### 2.5.2 Разработка базы нечётких правил

База правил формируется из двух источников: (1) статически заданные правила в коде и (2) записи в таблице DiagnosticDiagnosis с полем fuzzy_conditions.

**Примеры правил из миграции 0018_populate_diagnoses:**

1. **Повышенная тревожность** (код anxiety):
   - Условия: emotional_profile.грусть ≥ 0.4, motivational_potential.низкий ≥ 0.4.
   - Рекомендация: успокаивающие игры, дыхательные упражнения, консультация психолога.
   - Тип назначения: therapy.

2. **Импульсивность** (код impulsivity):
   - Условия: objectivity.низкая ≥ 0.5.
   - Рекомендация: упражнения на произвольное внимание и самоконтроль, игры с последовательным выполнением инструкций.
   - Тип назначения: exercise.

3. **Сниженная мотивация** (код low_motivation):
   - Условия: motivational_potential.низкий ≥ 0.6.
   - Рекомендация: короткие сессии с поощрением, постепенное увеличение сложности.
   - Тип назначения: recommendation.

4. **Эмоциональная нестабильность (гнев)** (код anger_elevated):
   - Условия: emotional_profile.гнев ≥ 0.35.
   - Рекомендация: релаксация, арт-терапия, техники управления гневом.
   - Тип назначения: therapy.

5. **Благоприятный профиль** (код favorable):
   - Условия: diagnostic_depth.высокая ≥ 0.5, motivational_potential.высокий ≥ 0.5.
   - Рекомендация: продолжать углублённую диагностику и мониторинг.
   - Тип назначения: recommendation.

**Примеры правил из fuzzy_logic.py (create_sample_rules):**

```python
{
    'name': 'Высокая тревожность',
    'condition': {
        'emotional_profile.грусть': '>0.5',
        'motivational_potential.низкий': '>0.4'
    },
    'conclusion': 'Повышенный уровень тревожности',
    'recommendation': 'Рекомендуются успокаивающие игры, консультация психолога',
    'confidence': 0.8
},
{
    'name': 'Импульсивный стиль',
    'condition': {
        'diagnostic_profile.cognitive_style': '== impulsive',
        'behavioral_patterns.высокий_импульсивность': '>0.6'
    },
    ...
},
{
    'name': 'Высокий диагностический потенциал',
    'condition': {
        'diagnostic_depth.высокая': '>0.7',
        'dynamic_assessment.широкий': '>0.6'
    },
    ...
}
```

Метод `_match_diagnoses` перебирает все записи DiagnosticDiagnosis, проверяет выполнение условий по порогам и возвращает список (диагноз, степень соответствия), отсортированный по убыванию соответствия и приоритету.

### 2.5.3 Дефаззификация и формирование заключения

Дефаззификация в классическом виде (центроид) реализована в `FuzzyVariable.defuzzify_centroid` для случаев, когда требуется получить одно числовое значение из набора принадлежностей. В диагностическом контуре основная задача — формирование текстового заключения и рекомендаций.

**Алгоритм формирования заключения:**

1. Сбор и расчёт лингвистических переменных A–E по результатам игр (calculate_diagnostic_depth, calculate_motivation, calculate_objectivity, calculate_ecological_validity, calculate_dynamic_assessment).

2. Определение когнитивного стиля (analyze_strategy): systematic, impulsive, adaptive, chaotic — на основе паттернов ошибок (mistake_types), типов игр (Sequence, Puzzle, Memory, GoNoGo) и performance_metrics.

3. Анализ эмоционального профиля (analyze_emotions): нормализация сумм эмоций по всем результатам. Выявление трендов (detect_emotional_trends) — increasing/decreasing/stable по половинам временного ряда.

4. Сопоставление с диагнозами (_match_diagnoses): для каждого диагноза проверяются условия; при выполнении диагноз попадает в detected_diagnoses, а его default_recommendations и default_prescription_text включаются в итоговый блок.

5. Генерация граничных рекомендаций (_generate_boundary_analysis): при превышении порогов по грусти (≥0.5, ≥0.35), гневу (≥0.5, ≥0.3), скуке (≥0.5, ≥0.35), низкой мотивации (≥0.6) добавляются соответствующие блоки с клиническими рекомендациями и возможными назначениями.

6. Дополнительные рекомендации (_generate_extra_recommendations): общие советы по диагностической глубине, мотивации, когнитивному стилю и эмоциям.

7. Формирование радарной диаграммы (get_radar_data): для каждой оси вычисляется взвешенная оценка по формуле (low×15 + mid×50 + high×95) / (low + mid + high), что даёт шкалу 0–100 для визуализации.

Итоговое заключение представляется в виде структурированного текста с разделами по выявленным диагнозам, граничным значениям и дополнительным рекомендациям. При отсутствии срабатывания правил выводится сообщение о норме и рекомендации продолжать мониторинг.

## 2.6 Проектирование пользовательских интерфейсов (по ролям)

### Интерфейс администратора

Главная страница — дашборд (`admin_dashboard.html`) с карточками: общее число пользователей, врачей, родителей, детей. Таблица пользователей с фильтрацией по роли, поиском и сортировкой. Блок непроверенных лицензий и последних регистраций. Навигация: Панель, Лицензии, Назначения. Страница проверки лицензий — список с фильтрами (ожидает/проверено), пагинация. Детальная страница лицензии — форма подтверждения/отклонения с указанием причины. Массовое назначение — две формы: привязка детей к родителю и пациентов к врачу. Статистика — графики (Chart.js): по типам игр, по эмоциям, по дням.

### Интерфейс врача

Панель врача (`doctor_dashboard.html`) — список пациентов с поиском и сортировкой. Форма добавления пациента по коду. Карточка пациента (`patient_detail.html`) — табы: «Диагностическая панель», «Результаты игр», «Назначения», «Профиль». Фильтр по датам и типу игры. В диагностической панели: радарная диаграмма (5 переменных A–E), сравнение с эталонами традиционных и цифровых методов; блоки функций принадлежности по параметрам (импульсивность, познавательная активность, стратегия, когнитивный контроль, тревожность); тепловая карта по системам (когнитивная, эмоциональная, поведенческая, регуляторная, социальная); график динамики; корреляционная матрица; адаптивные рекомендации и автогенерируемый текст назначения. Форма создания назначения встроена в карточку пациента. Страница углублённого анализа (`doctor_analysis.html`) — динамика эмоций по датам, траектории поведения, радар.

### Интерфейс родителя

Панель родителя (`parent_dashboard.html`) — список детей с краткой статистикой (количество игр, последняя игра, эмоциональный профиль). Форма подключения ребёнка по коду. Карточка ребёнка (`child_detail_parent.html`) — агрегированные проценты эмоций (радость, грусть, гнев, спокойствие), количество сыгранных игр, список активных назначений врача. Кнопки скачивания назначений в PDF/TXT. Упрощённый профиль (без детальных диагностических данных).

### Интерфейс ребёнка

Игровая панель (`game_dashboard.html`) — сетка карточек игр с названием, кратким описанием и кнопкой «Играть». Блок с кодом подключения (если есть) и кнопкой копирования. Достижения (например, «Сыграно 10 игр»). Страницы игр наследуют base.html и содержат игровую логику (HTML/JS), форму завершения или отправку данных через POST/API. Визуальный стиль — простой, крупные элементы, ориентирован на детей.

### Общие элементы

Базовый шаблон `base.html`: навигационная панель с логотипом, пунктами меню по роли, выпадающим меню пользователя (Профиль, Смена пароля, Выход). Подключены Bootstrap 5, Font Awesome, Chart.js. Адаптивная вёрстка для мобильных устройств.

### Десктопный клиент

Окно настроек (CustomTkinter): поле «Адрес сервера», кнопки «Проверить подключение» и «Войти». После входа — окно PyWebView с загрузкой страницы логина. Далее отображается веб-интерфейс системы в полноэкранном режиме (1280×800, min 1000×700).

## 2.7 Реализация компонентов системы (по модулям)

### Модуль аутентификации и авторизации

Реализован через Django-сессии. `login_view` проверяет пару username/password, при успехе записывает в сессию user_id, user_role, user_name и выполняет redirect на дашборд по роли. `logout_view` очищает сессию. Для API используется модель ApiToken: после успешного api_login создаётся токен, передаваемый в заголовке X-Auth-Token.

### Модуль управления пользователями

Формы: LoginForm, UserCreateForm, DoctorRegistrationForm, ParentRegistrationForm, ChildRegistrationForm. DoctorRegistrationForm при сохранении создаёт запись DoctorLicense. ConnectionCodeForm проверяет код ребёнка/врача по connection_code и code_expires. BulkChildAssignForm и BulkDoctorAssignForm — для массовых назначений.

### Модуль игр

Каждая игра имеет view для отображения (GET) и, при необходимости, view для сохранения результатов (POST). Например: game_painting_view, game_painting_save_view; game_choice_view, game_choice_save_view; game_dialog_view, game_dialog_save_view. Игры Memory, Puzzle, Sequence и др. могут сохранять результаты через встроенные формы или JavaScript. В GameResult сохраняются эмоции, reaction_times, mistakes, performance_metrics, drawing_data, dialog_answers, choices в зависимости от типа игры.

### Модуль нечёткой логики

FuzzyAnalyzer создаёт диагностический профиль методом create_diagnostic_profile(child_id). Внутри вызываются расчёты лингвистических переменных, analyze_strategy, analyze_emotions, detect_emotional_trends, generate_recommendations_with_diagnoses. Результат обновляет или создаёт запись DiagnosticProfile.

### Модуль диагностической панели

diagnostic_panel.py: get_fuzzy_analysis_for_panel (метрики, принадлежности, интерпретации), get_heatmap_data (матрица систем×показателей), get_dynamics_data (график по датам с трендом), get_correlation_matrix (корреляции эмоций и метрик), get_adaptive_recommendations (рекомендации по параметрам), build_auto_prescription_text (текст назначения для родителя). Используются справочники DIAGNOSTIC_PARAMETERS, MF_CLINICAL_GUIDE, BASE_RECOMMENDATIONS.

### Модуль REST API

api_views.py: api_login (POST), api_doctor_patients (GET), api_doctor_add_patient (POST), api_doctor_patient_detail (GET), api_doctor_create_prescription (POST), api_doctor_profile (GET), api_doctor_profile_update (POST/PUT/PATCH). Авторизация через _get_doctor_from_request (проверка X-Auth-Token). Ответы в формате JSON.

### Модуль экспорта и отчётов

parent_download_prescriptions_view и parent_download_prescription_view формируют PDF через ReportLab или текстовый файл при отсутствии ReportLab. export_patient_data_view выгружает данные пациента в удобном формате для врача.

Файл `accounts/tests.py` содержит заготовку для модульных тестов Django (TestCase). На момент анализа проектом не предусмотрены автоматические unit- или integration-тесты. Ниже описана логика проверки функциональности, которую целесообразно реализовать и/или выполнять вручную.

### Функциональное тестирование по ролям

**Администратор:**
- Вход под учётной записью admin и переход на дашборд.
- Фильтрация, поиск и сортировка пользователей.
- Редактирование пользователя, родителя, врача.
- Удаление пользователя (кроме admin).
- Просмотр списка лицензий, подтверждение и отклонение лицензии с указанием причины.
- Массовое назначение детей родителям и пациентов врачам.
- Просмотр статистики.
- Вызов инициализации нечёткой системы.

**Врач:**
- Вход и переход на панель врача.
- Проверка отображения сообщения при непроверенной лицензии.
- Добавление пациента по коду.
- Открытие карточки пациента, применение фильтров по дате и типу игры.
- Просмотр диагностической панели: радар, функции принадлежности, тепловая карта, динамика, корреляции.
- Создание назначения.
- Переход к углублённому анализу и экспорту данных.

**Родитель:**
- Вход и переход на панель родителя.
- Подключение ребёнка по коду.
- Просмотр карточки ребёнка (агрегированная статистика, назначения).
- Скачивание назначений в PDF/TXT.

**Ребёнок:**
- Вход и переход на игровую панель.
- Отображение кода подключения.
- Запуск каждой игры, прохождение до конца, сохранение результатов.
- Проверка появления результатов в карточке пациента у врача.

### Тестирование нечёткой логики

- Проверка фаззификации: при заданных reaction_times убедиться, что импульсивность попадает в ожидаемые термы.
- Проверка сопоставления диагнозов: при искусственно заданном профиле (например, высокая грусть, низкая мотивация) убедиться, что срабатывает правило «Повышенная тревожность».
- Проверка формирования рекомендаций: наличие блоков при граничных значениях (грусть ≥ 0.5, гнев ≥ 0.5 и т.д.).
- Проверка радарной диаграммы: корректность пересчёта get_radar_data при разных значениях термов.

### Тестирование API

- POST /api/auth/login с корректными и некорректными данными.
- GET /api/doctor/patients с токеном и без.
- POST /api/doctor/patients/add с кодом ребёнка.
- GET /api/doctor/patient/\<id\>/ с проверкой доступа только к своим пациентам.
- POST /api/doctor/patient/\<id\>/prescription/ на создание назначения.

### Тестирование десктопного клиента

- Запуск с флагом --launcher: отображение окна настроек.
- Проверка подключения при доступном и недоступном сервере.
- Вход: переход на страницу логина в PyWebView и отображение веб-интерфейса после аутентификации.
- Сохранение и загрузка настроек (app_config.json).

### Нагрузочное и интеграционное тестирование

При необходимости: симуляция множественных одновременных сессий, проверка производительности при большом количестве игровых результатов и создании диагностических профилей. Проверка работы на production-конфигурации (PostgreSQL, Gunicorn).
