## 1. Общая архитектура системы

### 1.1 Тип приложения

Система реализована в **гибридной архитектуре**:
- **Веб-приложение** — основной режим работы: доступ через браузер, хостинг на сервере;
- **Десктопный клиент** — нативное приложение для Windows: экран настроек (CustomTkinter), веб-интерфейс в окне PyWebView.

Оба варианта используют **единый серверный backend** и единый веб-интерфейс (HTML/CSS/JS), что исключает дублирование логики.

### 1.2 Архитектурный стиль

- **Клиент-сервер**: клиент (браузер или PyWebView) общается с Django-сервером по HTTP/HTTPS.
- **MVC-подобная структура (MTV в Django)**:
  - **Model** — модели данных (accounts.models);
  - **View** — представления и REST API (accounts.views, accounts.api_views);
  - **Template** — HTML-шаблоны (accounts/templates, templates).

### 1.3 Схема взаимодействия компонентов

## 3. Описание ролей

### 3.1 Ребёнок (child)

**Назначение:** прохождение игровых модулей для сбора диагностических данных.

**Доступные функции:**
- Вход по логину/паролю;
- Панель игр (`game_dashboard`): выбор из 12 типов игр;
- Прохождение игр: Painting, Dialog, Choice, Memory, Puzzle, Sequence, EmotionFace, Attention, GoNoGo, Sort, Pattern, EmotionMatch;
- Просмотр своих достижений (количество игр, последняя игра);
- Генерация кода подключения для родителей;
- Редактирование профиля, смена пароля.

**Ограничения:**
- Доступ только к своим данным;
- Нет доступа к результатам анализа, назначениям, панели врача.

**Сценарии использования:**
1. Регистрация ребёнка → получение кода подключения;
2. Вход → выбор игры → прохождение → сохранение результата;
3. Добавление родителя по коду (родитель вводит код ребёнка);
4. Передача своего кода врачу для добавления в список пациентов.

---

### 3.2 Врач (doctor)

**Назначение:** работа с пациентами, просмотр диагностики, создание назначений.

**Доступные функции:**
- Панель пациентов: список детей, добавленных по коду;
- Карточка пациента: результаты игр, радарная диаграмма, эмоциональные показатели, выявленные диагнозы, диагностическая панель (heatmap, динамика, корреляции, адаптивные рекомендации);
- Создание назначений (лекарство, терапия, упражнение, рекомендация);
- Фильтрация по дате и типу игры;
- Регистрация с указанием лицензии (номер, скан);
- Редактирование лицензии при отклонении;
- Экспорт данных пациента в JSON;
- REST API для десктопного клиента: получение списка пациентов, детали пациента, создание назначений (по токену X-Auth-Token).

**Ограничения:**
- Работа только с пациентами, добавленными по коду или администратором;
- Доступ к полной диагностике после проверки лицензии администратором;
- API доступно только для роли doctor.

**Сценарии использования:**
1. Регистрация → указание лицензии → ожидание проверки;
2. Добавление пациента по коду ребёнка;
3. Просмотр карточки пациента → анализ радарной диаграммы и рекомендаций;
4. Создание назначения;
5. Использование десктопного клиента: логин по REST API, работа с пациентами офлайн.

---

### 3.3 Родитель (parent)

**Назначение:** мониторинг детей и просмотр назначений врача.

**Доступные функции:**
- Панель «Мои дети»: список детей, привязанных по коду;
- Карточка ребёнка: агрегированная статистика (радость, грусть, гнев, спокойствие), количество игр, назначения врача;
- Скачивание назначений (PDF/TXT);
- Добавление ребёнка по коду;
- Редактирование профиля.

**Ограничения:**
- Доступ только к своим детям;
- Нет доступа к детальным результатам игр, диагностической панели, выявленным диагнозам;
- Видит только агрегированные эмоциональные показатели и назначения.

**Сценарии использования:**
1. Регистрация → добавление ребёнка по коду;
2. Просмотр карточки ребёнка → проверка назначений;
3. Скачивание назначений в PDF для визита к врачу.

---

### 3.4 Администратор (admin)

**Назначение:** управление пользователями, проверка лицензий, статистика.

**Доступные функции:**
- Панель пользователей: список, фильтр по роли, поиск, сортировка;
- Редактирование пользователей (admin, doctor, parent, child);
- Привязка детей к родителям, пациентов к врачам;
- Массовое назначение (bulk assign);
- Проверка лицензий врачей (подтверждение/отклонение);
- Удаление пользователей (кроме admin);
- Статистика: игры, сессии, активность по дням, эмоциональные показатели;
- Инициализация системы нечёткой логики.

**Ограничения:**
- Нельзя удалить администратора;
- Не создаёт/не редактирует игровые результаты — только связи и метаданные.

**Сценарии использования:**
1. Проверка заявок на лицензию врачей;
2. Назначение детей родителям при отсутствии кода;
3. Назначение пациентов врачу;
4. Просмотр статистики использования системы.

## 4. Описание игровых модулей

### 4.1 Перечень игр

| Код | Название | Тип данных |
|-----|---------|------------|
| Painting | Раскраска | Эмоции, drawing_data, color_counts, final_image |
| Dialog | Эмоциональный диалог | dialog_answers, эмоции, reaction_times |
| Choice | Эмоциональные сценарии | choices (раунды), эмоции, reaction_times |
| Memory | Память | pairs_found, attempts, levels_completed, time_seconds |
| Puzzle | Головоломка | moves, completed |
| Sequence | Последовательность | level_reached, mistakes |
| EmotionFace | Узнай эмоцию | correct, total, reaction_times |
| Attention | Внимание | hits, misses, false_alarms, reaction_times |
| GoNoGo | Стоп-игра | correct_go, correct_nogo, commission_errors, omission_errors |
| Sort | Сортировка | correct, total, reaction_times |
| Pattern | Паттерн | correct, total, reaction_times |
| EmotionMatch | Эмоция и ситуация | correct, total, choices, эмоции |

### 4.2 Логика работы и собираемые параметры

**Painting:** Ребёнок раскрашивает области. Собираются: цвета по областям, агрегация в эмоции (красный/оранжевый → anger; жёлтый → joy; зелёный → happiness; синий → sorrow; фиолетовый → love), изображение в base64, reaction_times (при наличии).

**Dialog:** Вопросы с выбором эмоций (joy, sorrow, love, anger, boredom, happiness). Собираются: ответы по вопросам (q1–q5), агрегация в счётчики эмоций, reaction_times.

**Choice:** Несколько раундов выбора изображений по эмоциям. Собираются: choices по раундам, reaction_times.

**Memory:** Игра «найди пары». Собираются: pairs_found, attempts, levels_completed, time_seconds.

**Puzzle:** Сбор картинки по порядку. Собираются: moves, completed.

**Sequence:** Повторение паттерна. Собираются: level_reached, mistakes.

**EmotionFace, Sort, Pattern, EmotionMatch:** Общий формат: correct, total, reaction_times, mistakes = total - correct.

**Attention:** hits, misses, false_alarms, reaction_times.

**GoNoGo:** correct_go, correct_nogo, commission_errors, omission_errors; mistake_types = {inhibition: commission_errors, attention: omission_errors}.

### 4.3 Формат передачи данных в модуль оценки

Результаты сохраняются в модели `GameResult`:
- `user` — ребёнок;
- `session` — ссылка на GameSession (опционально);
- `game_type` — код игры;
- Эмоциональные поля: joy, sorrow, love, anger, boredom, happiness;
- `reaction_time`, `reaction_times`, `mistakes`, `mistake_types`, `hints_used`, `hint_timing`;
- `accuracy`, `performance_metrics`, `strategy_type`;
- `fuzzy_analysis`, `drawing_data`, `dialog_answers`, `choices`, `final_image`.

Класс `FuzzyAnalyzer` получает список `GameResult` и формирует диагностический профиль через методы:
- `calculate_diagnostic_depth`, `calculate_motivation`, `calculate_objectivity`, `calculate_ecological_validity`, `calculate_dynamic_assessment`;
- `analyze_emotions`, `analyze_strategy`, `analyze_error_patterns`, `analyze_reaction_times`, `analyze_hint_usage`.

## 5. Описание алгоритма нечёткой логики

### 5.1 Входные переменные

**Основные лингвистические переменные (A–E для радарной диаграммы):**
- **diagnostic_depth** (A) — диагностическая глубина;
- **motivational_potential** (B) — мотивационный потенциал;
- **objectivity** (C) — объективность и стандартизация;
- **ecological_validity** (D) — экологическая валидность;
- **dynamic_assessment** (E) — потенциал для динамической оценки.

**Дополнительные:**
- **impulsivity** — импульсивность (по времени реакции, мс);
- **cognitive_activity** — познавательная активность (частота подсказок).

### 5.2 Диапазоны значений

| Переменная | Диапазон | Термы |
|------------|----------|-------|
| diagnostic_depth | [0, 1] | низкая, средняя, высокая |
| motivational_potential | [0, 1] | низкий, умеренный, высокий |
| objectivity | [0, 1] | низкая, средняя, высокая |
| ecological_validity | [0, 1] | низкая, средняя, высокая |
| dynamic_assessment | [0, 1] | ограниченный, умеренный, широкий |
| impulsivity | [0, 2000] мс | низкий, средний, высокий |
| cognitive_activity | [0, 10] (норм. частота) | исследующая, направленная, зависимая |

### 5.3 Функции принадлежности

**Тип:** трапециевидные. Для треугольных задаются [a, b, c], интерпретируются как [a, b, b, c].

**Формула (трапеция [a,b,c,d]):**
```
μ(x) = 0,           x ≤ a или x ≥ d
μ(x) = (x-a)/(b-a), a < x < b
μ(x) = 1,           b ≤ x ≤ c
μ(x) = (d-x)/(d-c), c < x < d
```

**Параметры термов (из fuzzy_logic.py):**

```python
'diagnostic_depth': {
    'низкая': [0, 0, 0.3, 0.4],
    'средняя': [0.3, 0.5, 0.7, 0.8],
    'высокая': [0.7, 0.8, 1, 1]
},
'motivational_potential': {
    'низкий': [0, 0, 0.2, 0.3],
    'умеренный': [0.2, 0.4, 0.6, 0.7],
    'высокий': [0.6, 0.8, 1, 1]
},
'impulsivity': {
    'низкий': [0, 0, 300, 400],
    'средний': [300, 400, 600, 700],
    'высокий': [600, 800, 2000, 2000]
},
# и т.д.
```

### 5.4 База правил (DiagnosticDiagnosis + FuzzyInferenceRule)

**Диагнозы (DiagnosticDiagnosis, fuzzy_conditions):**
- anxiety: emotional_profile.грусть ≥ 0.4, motivational_potential.низкий ≥ 0.4;
- impulsivity: objectivity.низкая ≥ 0.5;
- low_motivation: motivational_potential.низкий ≥ 0.6;
- anger_elevated: emotional_profile.гнев ≥ 0.35;
- sadness_dominant: emotional_profile.грусть ≥ 0.35;
- boredom_high: emotional_profile.скука ≥ 0.35;
- favorable: diagnostic_depth.высокая ≥ 0.5, motivational_potential.высокий ≥ 0.5.

**Правила вывода (FuzzyInferenceRule):**
- Высокая тревожность: emotional_profile.грусть > 0.5, motivational_potential.низкий > 0.4;
- Импульсивный стиль: cognitive_style == impulsive, behavioral_patterns.высокий_импульсивность > 0.6;
- Высокий диагностический потенциал: diagnostic_depth.высокая > 0.7, dynamic_assessment.широкий > 0.6;
- Систематическая стратегия: classification.систематический > 0.7, error_rate < 0.1;
- Случайный паттерн: classification.случайный > 0.6, error_rate > 0.3.

### 5.5 Метод нечёткого вывода

- **Фаззификация:** для каждой переменной вычисляются степени принадлежности к термам через `FuzzyVariable.fuzzify(value)`.
- **Агрегация условий:** для правил используется минимум по условиям (AND). Для диагнозов — проверка порогов: если все условия выполнены (val ≥ threshold), диагноз добавляется.
- **Активация заключений:** рекомендации формируются в `_generate_boundary_analysis` и `_generate_extra_recommendations` по граничным значениям эмоций, мотивации, когнитивного стиля.

### 5.6 Метод дефаззификации

Для радарной диаграммы используется **взвешенная комбинация термов** (не centroid):
```python
score = (low * 15 + mid * 50 + high * 95) / total
```
где low, mid, high — степени принадлежности к соответствующим термам; total = low + mid + high.

Метод `defuzzify_centroid` для общих случаев: центр тяжести трапеций с весами по membership.

### 5.7 Пример расчёта

**Вход:** 3 результата Memory (pairs_found=10,15,18; levels_completed=2,3,4), 2 результата Painting с drawing_data.

**depth_score:** diversity=5/12, has_detailed≈0.7, sessions_score=5/8 → depth_score ≈ 0.35×0.42 + 0.4×0.7 + 0.25×0.625 ≈ 0.56.  
Фаззификация: низкая=0, средняя≈0.8, высокая≈0.1.

**motivation_score:** memory_score, puzzle_score, completion_rate → пусть ≈ 0.65.  
Фаззификация: низкий=0, умеренный≈0.3, высокий≈0.7.

**Радар:** diagnostic_depth ≈ 50, motivational_potential ≈ 75 и т.д.

**Диагнозы:** если emotional_profile.грусть = 0.45, motivational_potential.низкий = 0.5 → anxiety срабатывает.

## 9. Тестирование

### 9.1 Виды тестирования

- **Ручное:** проверка сценариев по ролям (логин, регистрация, игры, назначения, лицензии).  
- **Unit-тесты:** `accounts/tests.py` — заглушка (TestCase без реализованных тестов).  
- **Инициализация нечёткой логики:** `python manage.py init_fuzzy_data` — создание переменных, правил, паттернов.

### 9.2 Проверка корректности нечёткого вывода

- Ручная проверка на наборах GameResult с известными характеристиками;  
- Проверка граничных значений: при error_rate 0.05 → систематический; при 0.35 → случайный;  
- Проверка эмоций: при грусть ≥ 0.5 → блок «Выраженная грусть» в рекомендациях;  
- Проверка диагнозов: при выполнении fuzzy_conditions код диагноза попадает в `detected_diagnoses`.

### 9.3 Тестовые сценарии

1. **Пустые данные:** create_diagnostic_profile при отсутствии GameResult → пустой/дефолтный профиль, рекомендация «недостаточно данных».  
2. **Только Painting:** расчёт diagnostic_depth, ecological_validity; объективность ниже, чем при наличии Memory/Sequence.  
3. **Только Memory/Puzzle/Sequence:** высокая объективность, динамическая оценка при наличии levels_completed.  
4. **Высокий гнев:** emotional_profile.гнев ≥ 0.5 → блок «Выраженный гнев/стресс».  
5. **Импульсивность:** много commission_errors в GoNoGo → strategy_type = impulsive, рекомендации по самоконтролю.

### 9.4 Граничные случаи

- Нулевые reaction_times → avoid division by zero, default 0.5 для impulsivity;  
- Один GameResult → недостаточно для трендов эмоций (detect_emotional_trends);  
- Все эмоции нулевые → нормализация даёт 0, рекомендации по умолчанию;  
- Истёкший connection_code → api_views возвращает «Срок действия кода истёк».
