## 2.1 Анализ технического задания и выбор средств разработки

Техническое задание на разработку программной системы предполагает создание веб-приложения и десктопного клиента для оценки когнитивных и эмоциональных процессов у детей дошкольного и младшего школьного возраста с использованием игровых методик [2, 3] и методов нечёткой логики [1]. Ниже приведён обоснованный выбор технологического стека в соответствии с требованиями к программной документации [5].

**Ядро серверной части.** В качестве веб-фреймворка выбран **Django** (версия 4.2 и выше) [14]. Причина выбора: встроенная ORM для работы с базой данных, система миграций, механизм сессий и аутентификации, развитая экосистема расширений и соответствие требованиям масштабируемости. Django обеспечивает строгую структуру приложения (MVC/MVT), что облегчает сопровождение и развитие системы.

**База данных.** В режиме разработки используется **SQLite** — встроенная СУБД, не требующая отдельной установки; для production предусмотрена **PostgreSQL** через расширение dj-database-url. Выбор обусловлен надёжностью, поддержкой транзакций и JSON-полей, необходимыми для хранения траекторий поведения и нечётких показателей [6].

**Математические расчёты.** Библиотека **NumPy** [16] применяется для вычисления средних, стандартных отклонений, коэффициентов корреляции и трапециевидных функций принадлежности. NumPy обеспечивает эффективную работу с массивами, что критично при обработке серий времени реакции и эмоциональных метрик [10].

**Обработка изображений.** **Pillow** используется для сохранения рисунков ребёнка (игра «Раскраска»), генерации превью и работы с медиафайлами. Выбор Pillow обусловлен широкой поддержкой форматов и интеграцией с Django (ImageField).

**Генерация PDF.** Библиотека **ReportLab** применяется для формирования PDF-отчётов с назначениями врача для родителей. ReportLab позволяет создавать структурированные документы с поддержкой кириллицы и произвольной вёрсткой.

**Развёртывание.** Для production-режима используются **Gunicorn** (WSGI-сервер) и **WhiteNoise** (раздача статических файлов без отдельного веб-сервера). Данная связка обеспечивает высокую производительность и простоту деплоя на облачных платформах (Render.com).

**Десктопный клиент.** Приложение реализовано на **CustomTkinter** и **PyWebView**. CustomTkinter предоставляет современный графический интерфейс для экрана настройки (адрес сервера) и быстрого запуска. PyWebView отображает веб-интерфейс системы в нативном окне, что позволяет использовать единый код фронтенда (HTML/CSS/JS) для веб- и десктопной версий без дублирования логики [15].

**HTTP-клиент.** Библиотека **Requests** используется в десктопном клиенте для проверки доступности сервера и в API-взаимодействии. Выбор обусловлен простотой использования и надёжностью при работе с REST API.

**Визуализация.** На стороне веб-интерфейса применяются **Chart.js** (графики эмоций, радарные диаграммы), **Bootstrap 5** (адаптивная разметка) и **Font Awesome** (иконки). На стороне десктопного клиента — **Matplotlib** для построения графиков при необходимости.

Соответствие ролей и зон ответственности отражено в концепции «зоны ближайшего развития» [2], когда родитель и врач выступают опосредованными участниками диагностического процесса.

## 2.3 Разработка структуры программной системы

Структура системы построена по модульному принципу с чётким разделением ответственности в соответствии с рекомендациями по проектированию программного обеспечения [5, 11].

На рисунке 2.2 представлена диаграмма компонентов системы, где выделены основные блоки: веб-сервер (Django), модуль нечёткой логики, REST API, десктопный клиент, игровой модуль, база данных. Показаны интерфейсы взаимодействия между компонентами (HTTP, JSON, ORM). Десктопный клиент связывается с веб-сервером через REST API при авторизации и через встроенный браузер (PyWebView) для основного интерфейса.

**Рисунок 2.2** — Диаграмма компонентов системы

**1. Веб-сервер (Django).** Приложение `accounts` содержит модели данных, представления (views), формы, модуль нечёткой логики и диагностической панели. Маршрутизация настроена в `accounts/urls.py` и подключается через `my_project/urls.py`. Конфигурация проекта вынесена в `my_project/settings.py`.

**2. Модуль нечёткой логики.** Отдельный модуль `accounts/fuzzy_logic.py` реализует классы FuzzySet, FuzzyVariable и FuzzyAnalyzer, обеспечивающие фаззификацию, анализ и формирование рекомендаций [1]. Модуль `accounts/diagnostic_panel.py` дополняет визуализацию и интерпретацию диагностических параметров.

**3. REST API.** Подмодуль `accounts/api_views.py` предоставляет эндпоинты для десктопного клиента: аутентификация по токену, получение списка пациентов, детальной информации о пациенте, создание назначений. API использует заголовок `X-Auth-Token` для авторизации врачей.

**4. Десктопный клиент.** Каталог `desktop_client/` содержит `app.py` (единая точка входа: экран настроек, проверка соединения, запуск PyWebView), `main.py` (обёртка для bat-файла), клиенты по ролям при необходимости.

**5. Игровой модуль.** Каждая игра реализована отдельным представлением и шаблоном. Результаты сохраняются в `GameResult` и `GameSession`. Метрики: эмоции, время реакции, количество ошибок, performance_metrics.

**6. Деплой.** Конфигурация для Render.com задаётся в `render.yaml`. Сборка десктопного приложения — через `build_desktop.bat` (PyInstaller).

## 2.4 Проектирование базы данных

Схема базы данных построена на моделях Django ORM [14]. Ниже приведено описание основных таблиц и их назначение.

На рисунке 2.3 представлена ER-диаграмма базы данных. Показаны сущности: CUsers (с атрибутами username, name, date_of_b, role, password, connection_code, code_expires), DoctorLicense, GameSession, GameResult, Prescription, DiagnosticProfile, DiagnosticDiagnosis, FuzzyLinguisticVariable, FuzzyInferenceRule, ApiToken, Subscription. Связи: CUsers — DoctorLicense (1:1), CUsers — GameResult (1:N), CUsers — children/patients (M:N), GameResult — GameSession (N:1), Prescription — CUsers (child, doctor), DiagnosticProfile — CUsers (child). Диаграмма отражает ключевые внешние ключи и кардинальности связей.

**Рисунок 2.3** — ER-диаграмма базы данных

### 2.4.1 Основные сущности

**CUsers** (Пользователи). Центральная таблица: username, name, date_of_b, role (admin/doctor/parent/child), password (хешированный), is_auth, created_at. Связи: ManyToMany children (родитель–дети), ManyToMany patients (врач–пациенты). Дополнительно: connection_code, code_expires.

**DoctorLicense** (Лицензии врачей). Связь OneToOne с CUsers. Поля: license_number, license_file, license_scan, is_verified, verified_by, verified_date, rejection_reason.

**GameResult** (Результаты игр). Связи: user, session. Поля: эмоции (joy, sorrow, love, anger, boredom, happiness), reaction_time, reaction_times (JSON), mistakes, mistake_types (JSON), hints_used, hint_timing (JSON), strategy_type, accuracy, performance_metrics (JSON), fuzzy_analysis (JSON), drawing_data, final_image, dialog_answers, choices.

**Prescription** (Назначения). Связи: child, doctor. Поля: text, prescription_type, medication_name, dosage, duration, date_created, is_active.

### 2.4.2 Модели нечёткой логики

**FuzzyLinguisticVariable.** name, description, terms (JSON), purpose. **FuzzyMembershipFunction.** method_class, variable (FK), membership_values (JSON). **DiagnosticProfile.** child, diagnostic_depth, motivational_potential, objectivity, ecological_validity, dynamic_assessment (JSON), cognitive_style, emotional_profile (JSON), recommendations, detected_diagnoses (JSON), based_on_sessions (M2M). **DiagnosticDiagnosis.** name, code, fuzzy_conditions (JSON), default_recommendations, default_prescription_type, default_prescription_text, priority. **FuzzyInferenceRule.** name, condition (JSON), conclusion, recommendation, confidence. **BehaviorPattern.** name, pattern_type, description, fuzzy_sets (JSON), relevant_games (JSON).

### 2.4.3 Дополнительные сущности

**Subscription.** OneToOne с CUsers. Типы подписок, max_children, max_doctors, start_date, end_date, is_active, price. **ApiToken.** Связь с пользователем, token.

## 2.5 Разработка алгоритмов оценки с использованием нечёткой логики

Методы нечёткой логики [1, 10] применяются для интерпретации «размытых» диагностических показателей, когда чёткие пороги (например, «норма/патология») не отражают градации состояний. В детской психологии данный подход оправдан вариативностью возрастного развития [2, 3, 4].

### 2.5.1 Фаззификация входных игровых данных

Фаззификация — преобразование числовых показателей в степени принадлежности к лингвистическим термам. Используется трапециевидная функция принадлежности. Для параметров [a, b, c, d] (где a ≤ b ≤ c ≤ d) степень принадлежности μ(x) вычисляется по формуле:

$$\mu(x) = \max\left(0, \min\left(\frac{x-a}{b-a}, 1, \frac{d-x}{d-c}\right)\right)$$

при условии b ≠ a и d ≠ c. На участке [a, b] функция линейно возрастает от 0 до 1; на [b, c] равна 1; на [c, d] линейно убывает до 0. Для треугольной функции (b = c) формула эквивалентна.

**Входные переменные и термы:**

1. **Диагностическая глубина** (0–1): низкая [0,0,0.3,0.4], средняя [0.3,0.5,0.7,0.8], высокая [0.7,0.8,1,1].
2. **Мотивационный потенциал** (0–1): низкий, умеренный, высокий.
3. **Объективность** (0–1): низкая, средняя, высокая.
4. **Экологическая валидность** (0–1): низкая, средняя, высокая.
5. **Потенциал для динамической оценки** (0–1): ограниченный, умеренный, широкий.
6. **Уровень импульсивности** (мс): низкий [0,0,300,400], средний [300,400,600,700], высокий [600,800,2000,2000].
7. **Познавательная активность** (0–10): исследующая, направленная, зависимая.

**Формулы для интегральных показателей A–E:**

Диагностическая глубина (A):
$$S_A = 0{,}35 \cdot \frac{|G|}{12} + 0{,}4 \cdot \frac{1}{n}\sum_{i=1}^{n} \sigma_i + 0{,}25 \cdot \min\left(\frac{n}{8}, 1\right)$$

где |G| — число различных типов игр, σ_i — полнота данных i-й игры (0–1), n — количество результатов.

Мотивационный потенциал (B):
$$S_B = 0{,}2 \cdot r_{compl} + 0{,}25 \cdot S_{mem} + 0{,}2 \cdot S_{puz} + 0{,}2 \cdot S_{seq} + 0{,}15 \cdot S_{emot}$$

где r_compl — доля завершённых сессий, S_mem, S_puz, S_seq — оценки по играм Memory, Puzzle, Sequence, S_emot — нормализованная сумма joy+happiness.

Объективность (C): S_C = 0,6 + 0,3·(N_obj/N) + 0,1·I(N_subj>0), где N_obj — число объективных игр, N_subj — субъективных.

Экологическая валидность (D): S_D = 0,6 + 0,1·(наличие Choice) + 0,1·(наличие Painting) + 0,1·(наличие Dialog).

Динамическая оценка (E): S_E = 0,35·I(n≥3) + 0,4·min(|G|/12,1) + 0,25·S_progress, где S_progress — доля прогресса по уровням.

### 2.5.3 Дефаззификация и формирование заключения

Для получения числового значения из принадлежностей используется метод центра тяжести (центроид) [1]:

$$x^* = \frac{\int x \cdot \mu(x) \, dx}{\int \mu(x) \, dx}$$

В дискретном представлении для трапецеидальных термов: для каждого терма вычисляются центр (b+c)/2 и площадь; взвешенная сумма даёт x*. В диагностическом контуре основная задача — формирование текстового заключения; числовая дефаззификация применяется для радарной диаграммы.

Формула для радарной оси (шкала 0–100):
$$V = \frac{\mu_{low} \cdot 15 + \mu_{mid} \cdot 50 + \mu_{high} \cdot 95}{\mu_{low} + \mu_{mid} + \mu_{high}}$$

Алгоритм заключения: (1) расчёт A–E, когнитивного стиля, эмоций; (2) сопоставление с диагнозами; (3) генерация граничных рекомендаций при превышении порогов; (4) дополнительные рекомендации. При отсутствии срабатываний — сообщение о норме.

## 2.9 Технико-экономическое обоснование

**Сравнение стоимости лицензий.** Разработанная система использует исключительно открытое программное обеспечение (Django, PostgreSQL, Python, NumPy, Pillow, ReportLab, CustomTkinter, PyWebView). Лицензионные расходы отсутствуют. Альтернатива на проприетарном стеке (например, .NET + MS SQL + коммерческие компоненты) потребовала бы ежегодных затрат на подписки (порядка 50 000–150 000 руб. для малой организации).

**Оценка трудозатрат.** Разработка с выбранным стеком (Python/Django) позволила сократить время реализации за счёт: (1) готовых решений Django (ORM, админка, аутентификация); (2) богатой экосистемы пакетов PyPI; (3) простоты прототипирования. Оценка трудозатрат для аналогичной системы на Java (Spring) или C# (ASP.NET) была бы на 20–30% выше из-за большей начальной настройки [13, 17].

**Стоимость хостинга.** Render.com (free tier): веб-сервис и PostgreSQL бесплатны в пределах лимитов. Для production с повышенной нагрузкой ориентировочная стоимость — от $7/мес (web) + $7/мес (DB). Альтернативы: Heroku (от $7), VPS (от 300 руб/мес).

**Сравнение с альтернативными технологиями.** Python/Django предпочтителен для данного проекта благодаря: быстрой разработке, встроенной поддержке JSON/нестандартных полей, наличию NumPy для научных расчётов. PHP (Laravel) уступает в структурированности и экосистеме для аналитики. Java/C# — избыточны для масштаба проекта и увеличивают порог входа для сопровождения.
